    Learning from discussion

Since the start of this competition, I almost never looked at the public
notebooks in the code section because it was filled with "upvote bait"—
notebooks where people simply mix others' kernels to farm upvotes.
Additionally, there were one or more arrogant cheaters using multiple
accounts to perform "blind blending."

In contrast, the discussion section contained a lot of high-quality
content. I read every single post in the discussions in detail and drew
significant inspiration from them:


      Orig FE

@masayakawamata <https://www.kaggle.com/masayakawamata> shared in this
post <https://www.kaggle.com/competitions/playground-series-s5e12/
discussion/648186> that using statistical features from the original
dataset could lead to score improvements. This indeed increased my score
slightly; however, I didn't perform any aggressive Feature Engineering,
so I only attempted to use the most fundamental statistics.


      The weird dataset

@tilii7 <https://www.kaggle.com/tilii7> mentioned in this post <https://
www.kaggle.com/competitions/playground-series-s5e12/discussion/652262>
that this dataset is very strange. The organizers distorted the feature
distributions and removed decisive strong features, resulting in an
inconsistency between the Train and Test distributions. To be honest, at
that point, I even considered giving up on the competition.


      ID Shift

The turning point came with this post <https://www.kaggle.com/
competitions/playground-series-s5e12/discussion/659313> by
@laureanoarcanio <https://www.kaggle.com/laureanoarcanio> . I strongly
recommend everyone read this post as it was the pivotal moment of this
competition. He mentioned a phenomenon: using |id| as a feature
surprisingly boosts the LB score.

Furthermore, @alan1305 <https://www.kaggle.com/alan1305> pointed out:

    One possible reason could be the interaction of id and
    physical_activity_minutes_per_week. If we plot the rolling mean of
    physical_activity_minutes_per_week, it shifts to test distribution
    at the end of train dataset. Therefore, including id may help
    indicate which row is "closer" to test set.

I conducted an Adversarial Validation analysis (Micro-Scan of the Tail
Transition) and found that this phenomenon indeed exists: *the closer
the data is to the tail, the closer its distribution is to the test set.*

This is actually a serious case of ID leakage, but given the significant
shift between train and test, this finding was crucial. (I honestly
don't know why @alan1305 <https://www.kaggle.com/alan1305> only received
5 upvotes… This was clearly a game-changing finding, yet people were
busy upvoting public blind-blending notebooks…)


      A validation strategy

In this post <https://www.kaggle.com/competitions/playground-series-
s5e12/discussion/662501>, @masayakawamata <https://www.kaggle.com/
masayakawamata> proposed a new validation strategy. Since the tail data
better "represents" the test set, it makes sense to use the tail data
directly for CV. He also introduced sample weighting.


      Concept Shift

@siukeitin <https://www.kaggle.com/siukeitin> mentioned in this post
<https://www.kaggle.com/competitions/playground-series-s5e12/
discussion/663033> that simple sample weighting doesn't work well
because there is *Concept Shift*.

I also conducted experiments regarding this. More specifically, within
the training set, the more the data resembles the test set (low
covariate shift), the greater the concept shift becomes. I divided the
data into 10 bins and found that samples with the smallest covariate
shift had the largest discrepancy in label distribution, whereas the
original dataset did not suffer from this issue.

|CUTOFF_ID = best_id

cat_cols = train.select_dtypes(include=['object']).columns.tolist()
for col in cat_cols:
    le = LabelEncoder()
    train[col] = le.fit_transform(train[col].astype(str))

train_ids = train['id'].values
cutoff_mask = train_ids >= CUTOFF_ID

X_tail = train.loc[cutoff_mask].drop(columns=['id', 'diagnosed_diabetes'])

print(f"1. Training Discriminator: Head({len(X_head)}) vs Tail({len(X_tail)})...")

# Binning

agg = df_eval.groupby('bin', observed=False).agg({
    'true_label': ['count', 'mean'],
    'tail_similarity': 'mean'
})
agg.columns = ['count', 'target_mean', 'similarity_mean']

print(agg)

plt.figure(figsize=(12, 6))

for i, v in enumerate(agg['target_mean']):
    plt.text(i, v + 0.005, f"{v:.3f}", ha='center', va='bottom', fontsize=9, fontweight='bold')

plt.axhline(true_tail_mean, color='red', linestyle='--', linewidth=2, label=f'True Tail Mean ({true_tail_mean:.3f})')

plt.title(" Do 'Tail-like' Head samples have correct labels?", fontsize=14)
plt.xlabel("Similarity to Tail (0=Unlikely, 9=Indistinguishable from Tail)", fontsize=12)

    Modeling

Based on the validation strategy derived from the analysis above, I
trained a diverse set of models. The core philosophy was to force the
models to learn the distribution of the "Tail" (which resembles the Test
set) and the "Original" dataset (which holds the ground truth logic),
while suppressing the noise from the "Head" of the training data.

      Ensemble Strategy

Given the diverse nature of the models (Linear, GAM, NN, and various
GBDTs), I used *Bagged Hill Climbing* to find the optimal weights.

  *

    *Method:* Bagged Hill Climbing (50 bags, 2000 iterations per bag)
    optimizing for *Cutoff AUC*.

  *

    *Final Result:* The ensemble significantly improved the robustness.

      o *Ridge Regression (Positive)*: 0.70751 Cutoff AUC
      o *Bagged Hill Climbing (Best)*: *0.70771 Cutoff AUC*

The final submission was generated using the weights from the Bagged
Hill Climbing method.

    Some advice for beginners

I noticed that many participants in this competition appear to be new to
machine learning. As a beginner myself, I completely understand the
learning curve, so I’d like to share a few simple tips based on my
experience:

 1. *Conduct purposeful EDA.*
    The ultimate goal of Exploratory Data Analysis (EDA) is to guide
    your modeling strategy. If your data analysis doesn't provide
    insights for feature engineering or model selection—and is just
    about plotting pretty charts—then I believe it lacks real value.
    Always ask yourself: "How does this plot help me build a better model?"
 2. *Don't obsess over micro-tuning hyperparameters.*
    Spending too much time fine-tuning parameters is often meaningless.
    Changing the learning rate from |1e-4| to |1e-5| is rarely worth
    wasting a submission. The most efficient approach is to spend that
    time trying out diverse models or exploring new feature engineering
    ideas.

hope you win gold medals in your upcoming competitions!

Relax, my friend. To be honest, I don't have a profound background in ML
theory, so I'm not sure if I'm qualified to give you advice on "learning
ML." For a competition, understanding the data format and the rules
might actually be more important.

My post essentially answers just one question: Why did I choose these
models?
I included some models by borrowing ideas from the discussion section,
others simply because I noticed their CV scores were high, and some to
verify conclusions drawn from experiments. However, this approach is
specific to this competition and lacks generalizability. What I mean is,
if you enter a new competition with a new dataset, it likely won't have
specific quirks like "the tail data resembles the test distribution."

Therefore, when facing a new competition, you need to ask yourself: "Why
am I choosing this model?" As long as you figure this out, even if you
are a beginner, it is possible to achieve a good ranking.

Hi, thanks for your comment! This is a very interesting question.

    Could you please give some advice in the field of feature
    engineering for beginners like me?

For traditional ML competitions like this one, I think whether or not to
do FE depends on a few key factors:

 1. *Understanding* is the most important thing. Our models are actually
    “smarter” than us in some sense, so you must constantly try to
    understand /what the model is actually seeing/. Take this
    competition as an example: when I added |id| as a feature, both the
    CV score and the LB score improved. By common sense, |id| should not
    matter at all—it’s just a sample index. When something like this
    happens, you need to stop and think: /is my model capturing some
    hidden pattern that I haven’t noticed yet?/
    This observation eventually led people in the discussion to infer
    that the tail of the training data is closer to the test
    distribution, which essentially determined the entire modeling
    strategy afterward.
 2. Many of my feature engineering ideas came from *experiments*. In
    this notebook <https://www.kaggle.com/code/daylighth/ps-s5e12-
    hypothesis>, I recorded various experiments. For example, if we
    assume that the tail dataset can accurately represent the test set,
    Experiment 7 shows that a significant portion of features have
    strong linear relationships. This explains why simple models like
    logistic regression actually performed quite well in this competition.
 3. *Intuition*. It’s hard to clearly explain. Hmm… to be honest, most
    of my code was generated by AI(of course I know how to modify and
    validate it). AI can give you many suggestions about feature
    engineering or model selection, but you must decide which ideas are
    potentially valuable and which ones are just nonsense.
    There is no trick or strict standard to determine whether AI is
    lying, nor is there a formula to tell you "this approach is right,
    and that one is wrong." Therefore, I believe intuitive judgment
    plays a key role in decision-making.

Of course, everything I mentioned above still doesn’t explain everything
—after all, machine learning is largely an empirical discipline.
If you have many ideas but aren’t sure whether they’re correct,*JUST DO
IT*. Some ideas may turn out to be completely wrong, but that’s totally
fine—the code can always be changed. Even if you end up with a very low
score, at least you’ve learned that “this path doesn’t work.”
In fact, the models I shared here are not all the ones I tried. They are
only the ideas that worked. I experimented with many more feature
engineering approaches and models behind the scenes.

Happy New Year and have fun!
