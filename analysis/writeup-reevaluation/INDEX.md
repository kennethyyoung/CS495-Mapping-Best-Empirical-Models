## Cross-cutting observations

Patterns that show up across **multiple** writeups (N ≥ 2). Single-case observations stay in the per-writeup doc's *Surprising / unusual* section until a second case promotes them here. Cite the writeups each entry rests on.

### Constraint → viable strategies

Datasets and competition structures don't have a single "winning strategy" — but specific *constraints* appear to make specific *strategy families* viable (or even forced). A coupling enters this table when the same constraint→strategy pairing is observed in ≥2 cases. This table is the intended spine of the flowchart deliverable.

| Dataset / competition constraint | Viable strategy | Cases (N) |
|---|---|---|
| Synthetic data with known public original | Exploit-original toolkit: snap features (s6e3), KDTree nearest-neighbor lookup (s6e3), **exact row-match join** (s5e7 `match_p`), external target statistics / WoE / entropy, drift ratios, "original_as_columns" feature concat (s6e3, s5e6 explicit), original-as-rows concat (most cases), **TE on combinations × original-data-targets** (s5e6 scaled to 2268 cols; s5e11 with alt-target variant) | s6e4_rank1, s6e3_rank1, s6e2_rank1, s5e12_rank1, s5e11_rank1, s5e8_rank2, s5e7_rank2, s5e6_rank1 (N=8 of 8 applicable cases; s6e1 had no public original and the toolkit is absent there — consistent) |
| Heavy public-notebook coverage + author with framework | Fork-heavy ensembling: port 4+ named public notebooks as base models, train each many seeds, add author's own "monster model" on top, blend via HC / linear stacker | s6e3_rank1 (KGMON Playbook 2026: 39 ports + LLM-coded models), s5e6_rank1 (proto-KGMON: 4 named ports of 9 base models), s6e4_rank1 (60+ public OOFs across named contributors) (N=3). **Counterexample within author**: s5e5_rank1 — same author (cdeotte), no fork-heavy ensembling, builds substantially from his own techniques. Suggests fork-heaviness emerges *after* an author's starter notebooks form a community, not as a constant author trait. |
| Many OOFs at linear-stacker scale (>~50) | Linear stacker (Ridge / LogisticRegressionCV / cuML L2 LR / HC rank-blend / equal-weight mean blend) as **final** meta-learner — anti-overfit choice over nonlinear stackers (s5e11 explicit: "stacking with non-linear models... much worse than linear"). **Not universal:** s5e8 (Mahog, CatBoost), s4e9 (Mart Preusse, **DeepTables NN as final stacker** — first nonlinear), s4e11 (Mahdi Ravaghi, AutoGluon as final stacker), s4e7 (Cross Sellers, **XGBoost as final stacker** + author explicit "Linear approaches did not work"). **Doesn't apply at all to single-model winners.** | Linear: s6e4_rank1, s6e3_rank1, s6e2_rank1, s6e1_rank1, s5e12_rank1, s5e11_rank1, s5e10_rank1, s5e7_rank2, s5e6_rank1, s5e5_rank1, s5e3_rank2, s4e10_rank2 (HC), s4e8_rank1 (AG+Ridge variants) (N=13). - **Multiclass + rare class + balanced-accuracy metric → binary reframe** (split into one-vs-rest dichotomies). Observed: s6e4_rank1. Test on: future multiclass/imbalanced competitions, especially those with non-standard metrics.
